§5.1: Exec System Call
  • On a high level, the exec system call loads a program into memory and then executes it.
  • The GCC compiler can create an executable. We can also create object files using a flag in the command. With the help of a linker, we can turn these object files into an executable.
  • Suppose the ELF file is a.out. Then inside the shell we create a child process using fork(). Inside the child process we can execute a.out using the exec system call while the parent waits for the child to complete. The parent process is put into sleeping state until the child process is complete.
  • The compiler generates the files in ELF format and OS executes them.
§5.2: ELF Format of executables: Linker View and Executable View
  • The ELF format has two views - i) The Linker View ii) The Executable View.

The Linker View

  • The ELF header is common for both the linker view and the executable view. Starts with ELF and has an identifier. It further specifies the type of ELF file (executable, relocatable, shared object), machine details (i386, x86, arm) and the entry (virtual address where program begins execution), pointer to the program header, number of program headers, pointer to section header , number of section headers.
  • As an object file cannot be executed, the address corresponding to entry is set to 0. For an executable, the program headers are important whereas for an object, the section headers are important.

The Executable View

  • The program header contains the type of segment, offset of segment in ELF file, virtual address where the segment is to be loaded, physical address (ignored in OS based systems) where the segment is to be loaded, size in file image (size on hdd), size in memory(size when loaded onto memory) and flags.
  • When OS is creating a process using the exec system call, it would look into program headers and create virtual address space accordingly.
  • In the executable view there would also be a mapping between segments and sections.
§5.3: Exec implementation in xv6
  • The exec function takes in 2 parameters: path of the executable and command line arguments.
    1. Then the function converts the path to a pointer in the hdd called inode.
    1. Then ELF header is read using readi takes the the pointer/inode and reads sizeof(elf) bytes) and stores the result in elf.
    1. Then for sanity check the magic number (x7fELF) is matched.
    1. Then the virtual address space for the child process that invoked exec is identified and the pointer to its page table is stored in pagetable.
    1. Then we look into the program header table, parse through the elf program headers for each segment and load them into memory if they are of type LOAD and add more page table entries. The loadseg function copies the segment page by page from the harddisk to the memory at the specified address in the header.
    1. Then the user stack and guard pages are created.
  • argc (in a0 register, # of arguments) is in an integer and argv(in a1 register, arguments) is an array of pointers. The exec function is being executed from the kernels VA. However the arguments and the pointers need to be in childs VA. The function copyout copies these arguments from kernel VA to child VA.
